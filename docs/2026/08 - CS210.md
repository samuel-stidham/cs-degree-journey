**Course Description:**
Students will develop functional programs that comply with industry regulations and best practices using various programming languages. Special attention will be paid to the importance of developing code that is not only functional, but also secure, efficient, and professional.

## Key Topics & Concepts

This course builds C++ fluency from the ground up, then carries the same design habits across other languages:

* **C++ Language Fundamentals:** Writing explanatory comments, following team coding standards, and choosing appropriate data types (`int`, `string`, `char`, `double`) for efficient internal data storage — including cross-language quirks like C++'s lowercase `string` versus Java's capitalized `String`.
* **Control Flow & Debugging:** Building conditional logic with relational and logical operators, constructing loops to repeat code blocks, and stepping through a debugger to inspect program state and resolve logical errors.
* **Pre-Code Design:** Using UML class and use-case diagrams, pseudocode, and flowcharts as language-agnostic tools to plan a program's structure and behavior before writing any code.
* **Data Structures:** Distinguishing fixed-size arrays from dynamically resizable vectors for storing collections of same-typed values.
* **Functions:** Designing named, parameterized functions with defined return types and properly scoped local variables to perform discrete tasks.
* **Object-Oriented Programming in C++:** Defining classes with member variables and methods, controlling access through public, private, and protected modifiers, and applying the four OOP pillars — encapsulation, inheritance, polymorphism, and abstraction.
* **File Operations:** Using the `fstream` library to open, read, write, append, and close sequential files so a program can process data without requiring manual user input.
* **Exception Handling:** Using the try/catch/throw framework to detect runtime problems and let a program recover, report an error, or shut down gracefully instead of crashing.
* **Pointers & Memory Management:** Using pointers to reference and pass data by memory address rather than by value, and manually managing the allocation and deallocation of that memory.
* **Professional Standards & Ethics:** Following organizational style guides for naming, commenting, and formatting, and applying a professional code of ethics covering testing, attribution, licensing, confidentiality, and data protection. Built an initial GitHub portfolio repository to track project artifacts over the course of the degree.
* **Foundational Competencies:**
  * **CS-20401:** Utilize various programming languages to develop secure, efficient code.
  * **CS-20402:** Design functional programs that comply with industry regulations and best practices.
  * **CS-30403:** Write professional, well-documented, and readable code.

## Major Projects & Assignments

1. **Project One: Chada Tech Dual Clock**
    * **Focus:** A simulated technical interview exercise for "Chada Tech" — building a C++ console app that displays simultaneous 12-hour and 24-hour clocks (per ISO 8601) and lets the user advance the time via a menu.
    * **Approach:** Split the program into `Clock` (pass-by-reference time mutators and string formatters) and `Display` (bordered, side-by-side rendering and menu) modules, keeping `main()` to just state setup and a dispatch loop. Used `std::stoi` with exception handling for input validation instead of relying on `cin >>`, avoiding the stream-state pitfalls that approach creates on bad input.

2. **Project Two: Airgead Banking Investment Calculator**
    * **Focus:** An OOP client project for "Airgead Banking" — a console app that projects investment growth year by year under monthly compound interest, both with and without recurring monthly deposits.
    * **Approach:** Planned the program as pseudocode before writing any code, then split it into two classes: `InvestmentCalculator` (stores validated investment data and performs the monthly-interest calculation) and `InvestmentReport` (handles user I/O and renders both year-end tables). Used setters that throw `std::invalid_argument` on bad data rather than silently storing it, with the input-collection loop catching that exception and re-prompting until all four values are valid. Generated both required reports from the same year-end calculation method by simply toggling whether the monthly deposit is included, rather than duplicating the loop logic.

3. **Project Three: Corner Grocer Item Tracker**
    * **Focus:** A grocery analytics console app for "Corner Grocer" that reads a day's purchase log and reports item frequencies via search, a full alphabetical listing, and a text histogram — built around a single `GroceryTracker` class with public/private sections, backed by a `std::map<std::string, int>`.
    * **Senior-Level Judgment Calls (documented in-code):**
        * **Cognitive Complexity Refactor:** The menu loop originally tangled input validation, menu routing, and individual actions together in one method. Split it into `printMenu`, `readMenuChoice`, `dispatchChoice`, and `handleItemSearch` so a bug in one area only requires inspecting that specific method instead of untangling the whole control flow.
        * **Container Trade-off, Documented Rather Than Assumed:** Chose `std::map` over `std::unordered_map` for the frequency table — its ordered iteration gives alphabetical output for free, and the O(log n) lookup cost is irrelevant at this data volume. Noted in-code that at enterprise scale, where lookup speed matters more than free ordering, `std::unordered_map`'s O(1) average lookup would be the better trade despite losing sorted iteration — treating "most optimal" as a judgment call based on actual data volume rather than a default reflex.
    * **Other Notes:** `getFrequency` resolves an item in a single map lookup (a raw pointer to the found value, or `nullptr`) rather than checking existence and then looking the value up a second time. `frequency.dat` is written automatically at startup so the backup file is always in sync with the input log without requiring user action.

4. **GitHub Portfolio Submission**
    * **Focus:** Culmination of the course's portfolio component, split across two modules — Module Seven walks through creating a GitHub repository and adding the instructor as a collaborator; Module Eight is publishing one project (Project Three) to that repository so the instructor can grade that the files are present.
    * **Workflow Note:** The course's tutorial walked through uploading the project to GitHub manually through the web UI. Instead, I initialized a real local Git repository (`git init`) inside the Visual Studio project, wrote a proper `.gitignore` to exclude build artifacts and IDE-specific files, and pushed the project using an SSH key — the same workflow used professionally, rather than the simplified manual-upload process the course taught.

## Discussion Work & Academic Themes

1. **Introduction: Tell Us Who You Are**
    * **Focus:** Introduced professional background and explained why a working Senior Software Engineer with 13-14 years of experience was returning for a formal CS degree.
    * **Reflection:** Framed the degree as a way to close theoretical gaps left by an original Associate's degree in IT — particularly around choosing the right data structure for a problem rather than relying on intuition — citing Donald Knuth and Leslie Lamport as the standard being aimed for. Detailed existing C++ familiarity from prior coursework and a personal TUI roguelike project implementing Dijkstra's maps for pathfinding, along with broader language experience (PHP, Go, Rust, Python, Ruby, JavaScript, C#) and a professional preference for LLVM's C++ coding standards over Google's, citing their closer alignment with Stroustrup and Sutter's C++ Core Guidelines.

2. **Ethics: What Would You Do?**
    * **Focus:** Responded to a prompt on the ethics of refusing unethical coding requests, grounded in two real incidents from a 13+ year engineering career.
    * **Analysis:** Described refusing a request to falsify accounting figures to defraud a bank for a loan, and separately refusing a request to store credit card numbers, CVVs, and expiration dates in plain text in violation of PCI compliance. Mapped both incidents to specific clauses of the ACM/IEEE-CS Software Engineering Code of Ethics (ACM, 1999) — the accounting case to Clause 4.04 (deceptive financial practices) and Clause 6.06 (legal compliance), and the plaintext card storage case to Clause 3.12 (respecting privacy) and Clause 1.03 (avoiding harm to privacy).
    * **Takeaway:** Argued that citing a recognized professional code turns a refusal from a personal opinion into a defensible professional standard — protecting the end user, the engineer, and the organization, and giving the engineer grounds for recourse if retaliated against for refusing unethical work.

## Personal Reflection & Takeaways

This was effectively my third programming course covering C++ from the ground up, on top of roughly fourteen years of professional software engineering experience. As a result, the course content itself — comments and data types through a brief introduction to classes, with topics like templates left out — mostly reinforced knowledge I already had rather than introducing new material. The one exception was deliberate: writing up the cognitive-complexity refactor and the `std::map` versus `std::unordered_map` trade-off in Project Three's documentation. Both of those habits came from my career, not from this course, but documenting them clearly was a useful exercise in its own right.

Grading in this course followed a strict, checklist-style rubric with little apparent room for differing interpretations of an assignment. That occasionally felt disproportionate to the competency actually being demonstrated — for example, losing a meaningful number of points on a "find the bug" exercise despite correctly identifying the overwhelming majority of the bugs. I'd rather reserve final judgment until the course is fully complete, but for students with less programming background than mine, I'd expect that same rigidity to have an even larger impact on outcomes.
